# NeKI — Newsletter Knowledge Index

Use NeKI as a compact routing layer to third-party developer writing. Treat every entry as a perspective or lead, never as an API specification, guaranteed best practice, or replacement for current Apple documentation.

## Lookup

- Start with [references/README.md](references/README.md). Use a topic file for a complete cross-source view of all matching indexed reading, a newsletter file for curated external reading, or a blog file for a publication's own articles. Publication entries include concise NeKI-authored briefs only where enough specific context is available.
- Use [references/sources.md](references/sources.md) to find the publications currently represented by NeKI.
- Treat [references/candidate-domains.md](references/candidate-domains.md) as discovery material only: inclusion there is not an endorsement or a quality judgement.
- For framework/API facts, version availability, or normative guidance, verify the finding with Apple documentation (use ApKI where available).

## Source evaluation

Before relying on a third-party article, check its publication date, platform and SDK version, author context, cited primary sources, and fit for the current codebase. Prefer an article that demonstrates its trade-offs over one that merely presents a recipe. Do not elevate a repeatedly linked domain to a trusted source without an explicit quality review.
